# AGENTS.md


## Project Overview

本專案是「捷運路線規劃既各站資訊」的前端展示 Demo。
目標是做出一個可以在瀏覽器中打開的靜態網站，展示：
-車廂位置
-樓梯或電扶梯位置
-轉乘路線與時間
-鄰近出口位置
-特殊設施位置
-資料結構如何被使用
本專案主要用於課堂展示與期末報告，不是實際捷運資訊系統。


#Tech Stack

請只使用：
- HTML
- CSS
- JavaScript
- 不使用後端
- 不使用資料庫
- 不使用登入系統
- 不使用 npm / React / Vue 等框架
最終作品必須可以直接用 index.html 開啟。

## Must Have
* **車廂精確對應**：所有位置資訊必須精確至「車廂編號」（例如：第 3 節車廂）。
* **多條件路徑規劃**：必須同時提供「最快轉乘（走樓梯/電扶梯）」與「無障礙優先（全電梯動線）」兩種路徑。
* **量化轉乘動線**：轉乘路線必須包含精確的步行距離與預估時間（分鐘）。
* **出口與設施連動**：鄰近出口資訊必須與特定設施（如洗手間、詢問處、電梯）進行綁定與關聯。
* **動靜態資料分離**：站體結構（靜態）與設施維修/營運狀態（動態）必須分開儲存與查詢。

##Do not do
* **嚴禁模糊描述**：禁止使用「在月台中段」、「靠近前面」等模糊詞彙，無法量化的資訊不予顯示。
* **禁止硬編碼動態資訊**：首末班車時間、設施維修狀態等易變動資訊，嚴禁寫死在程式碼中，必須透過 API 讀取。
* **禁止資訊過載**：不要在單一畫面上塞滿所有站點設施，必須採用漸進式揭露，優先顯示核心路徑。
* **禁止混淆營運模式**：針對區間車與全程車、不同的開門方向，禁止混為一談，必須依據列車班次精準判斷。

## 驗收標準 (重要)

### 1. 基本執行

- 使用者可以直接打開 index.html 看到完整畫面
- 不需要安裝額外套件
- 不需要登入
- 不需要後端伺服器
- 不需要 API key
- 頁面載入後不能出現明顯錯誤畫面
- 瀏覽器 console 不應該有會影響功能的 JavaScript error

### 2. Demo 流程

使用者必須能完成一個完整展示流程：
1. 進入首頁。
2. 選擇或看到一個明確的情境。
3. 操作主要功能。
4. 系統產生明確結果。
5. 結果可以被截圖或口頭說明。
6. 整個流程可以在 3 分鐘內展示完。
如果是互動式作品，至少要有一條「保證可成功展示」的 demo path

### 3. 核心功能

- 至少完成 MVP 中定義的核心功能。
- 核心功能不能只是靜態文字描述，必須有可操作或可觀察的結果。
- 每個按鈕或主要互動元件都應該有明確作用。
- 不應存在點了沒有反應、但看起來像主要功能的按鈕。
- 如果功能尚未完成，請不要放在主要畫面上誤導使用者。

完成時必須能回答：
- 使用了哪一種資料結構？
- 資料是怎麼存的？
- 為什麼這裡適合用這種資料結構？
- 這個資料結構如何影響功能運作？

### 4. 檔案與程式碼

- 主要檔案命名清楚，例如 `index.html`, `style.css`, `script.js`。
- 不要把大量 JavaScript 全部塞在 HTML 裡。
- 不要留下大量沒用到的程式碼。
- 不要留下測試用假按鈕、假頁面或無意義文字。
- 函式名稱應該能看出用途。
- 如果有資料檔，資料結構要清楚，不要全部寫成難以閱讀的一大坨字串。

### 5. 穩定性

- 重新整理頁面後仍可以正常使用。
- 操作順序稍微不同時，不應直接壞掉。
- 沒有選資料就按按鈕時，系統應該給出提示，而不是報錯。
- 如果輸入錯誤，畫面應該有清楚回饋。
- Demo 前應至少完整跑過 3 次主要流程。

### 6. 範圍控制

以下內容不是必要項目，不應該影響 MVP 完成：
- 登入系統
- 後端資料庫
- 真實金流
- 真實通知系統
- 真實 GPS
- 複雜 AI 模型
- 大量頁面
- 過度精緻但無法操作的動畫

### 10. 最終完成定義

當以下條件都達成時，才算完成：
- 可以正常開啟作品。
- 可以完成主要 demo 流程。
- 有清楚的輸入、處理、輸出。
- 至少一個資料結構有被實際用在功能中。
- 畫面適合展示。
- 團隊能說明作品邏輯。
- 沒有明顯會讓 demo 中斷的錯誤。

補充 : 以上驗收過程建議使用 browser use 功能 (/browser) 做測試
